## What BestPhilippines is

A scored, ranked, living guide to where the Philippines eats, drinks, and stays out late — positioned as the Pitchfork of Asian dining, drinking, and clubbing.

Every venue has a number next to it (0.0–10.0, one decimal) generated by a published algorithm with a bounded editorial boost. No pay-to-play. No unlabeled promotion. No hidden gems.

We are currently in **Phase 1** (utility-first, algorithmic scoring, no named critics). See `/docs/product/spec-v2.md` for the full phasing.

## The hard rules — never violate

These are the project's non-negotiables. Code that breaks any of these is wrong code.

1. **Nothing affects the score except the algorithm and the bounded editorial boost.** Not ads. Not affiliates. Not paid boosts. Not partnerships. The score pipeline accepts inputs only from approved external data providers (see `/docs/decisions/0005-algorithm-weights.md` for the current list) and the editorial boost field (bounded ±1.0). No other code path writes to the score column.

2. **The 5.0 floor is absolute.** Venues scoring below 5.0 do not appear on the site. The listing query must filter `score >= 5.0`. No exceptions in Phase 1.

3. **The editorial boost is bounded at ±1.0.** Hard-coded constraint at the database level and the API level. Not configurable from the admin UI.

4. **Every editorial boost is logged.** When a boost is applied, an `editorial_boost_log` row is written with: venue_id, editor_id, boost_before, boost_after, reason, timestamp. No silent adjustments.

5. **Banned words never appear in microcopy, UI strings, or generated content.** See the full list in `/docs/editorial/voice-guide.md`. Partial list: *hidden gem, elevated, transcendent, must-try, authentic, foodie, amazing, incredible, awesome*. Do not write these anywhere a user can see them. A custom ESLint rule for the banned-words list will live at `eslint-rules/no-banned-words.js` and be enforced in CI. Until that lint exists, Claude Code should self-check any user-facing string against the voice guide before committing.

6. **Named critics do not exist in Phase 1.** Do not add critic profiles, bylines on algorithmic entries, or critic-specific UI. Lists are attributed to "Martin Casey," "Yahnee Ortiz," or "BestPhilippines editors" — never to invented names. If you see legacy names like "Kara Lim," "Mika Reyes," or "JP Cuizon" in seed data, they are drift from an earlier prototype and should be replaced.

7. **List titles reflect actual count.** A list of 6 bars is titled "The 6 bars worth your time in [neighborhood]," not "Top 10 bars in [neighborhood]." See `/docs/decisions/0004-list-title-convention.md`.

8. **No sub-5.0 published content in Phase 1.** This includes review drafts, preview pages, and admin-visible scores that get pushed to production unintentionally. Sub-5.0 named-critic long-form is a Phase 2 feature.

## Voice — the short version

Full guide at `/docs/editorial/voice-guide.md`. Shortest version:

- Specific beats superlative. *The sisig arrives hissing on cast iron,* not *amazing sisig.*
- Opinions held clearly. *Service was slow. Twice.* Not *some might find service slow.*
- Warmth baseline, sharpness as tool. Filipino generous by default; sharp when earned.
- Filipino without cosplay. No italics on Filipino dish names. No "exotic." No tourist register.
- Useful beats beautiful. Help readers decide where to eat tonight.
- **The test:** would a Filipino food writer you respect nod at this, or roll their eyes?

When generating microcopy, UI strings, error messages, or any user-facing text: apply the voice guide. When in doubt, err toward plain and specific.

## Tech stack

- **Framework:** Next.js (App Router) with TypeScript (strict mode)
- **Styling:** Tailwind CSS
- **Database:** Supabase (Postgres + Auth + Edge Functions)
- **Deployment:** Vercel
- **Cron:** Vercel Scheduled Functions (or Supabase Edge Functions for heavy batch)
- **Image handling:** Next.js Image component, with Supabase Storage or Cloudinary
- **External APIs:** Google Places (primary), TripAdvisor Content API (secondary), plus optional Foursquare/Facebook Places signal. FX rates via exchangerate.host. See `/docs/decisions/0005-algorithm-weights.md` for the canonical list and `/docs/decisions/0008-external-data-providers.md` for provider selection rationale.

## Coding conventions

- **TypeScript strict mode on.** No `any`. When a type is genuinely unknown at a boundary (e.g., third-party API responses, user input), use `unknown` and narrow explicitly with type guards or Zod parsing — not `any`. If narrowing is non-trivial, ask rather than bypass.
- **Server Components by default.** Use Client Components only when interaction requires it.
- **Database access via Supabase client only.** No raw SQL in app code except in Supabase migrations.
- **Server actions for mutations.** Not API routes unless explicitly warranted.
- **No localStorage / sessionStorage in the main app.** Use Supabase for persistence.
- **Naming:**
  - Components: PascalCase (`VenueCard.tsx`)
  - Utilities: camelCase (`computeScore.ts`)
  - Routes: kebab-case (`/about/scoring`)
  - Database tables: snake_case (`editorial_boost_log`)
- **File size limit:** if a component passes 300 lines, split it.
- **Test the algorithm and the boost logic.** Unit tests on the scoring math are non-negotiable. The rest of the app can be lightly tested.

## Database schema — core tables

Full schema in `/docs/product/data-pipeline.md`. Core tables for reference:

- `venues` — the master record
- `venue_scores` — current score components (algorithmic, boost, final)
- `score_history` — weekly snapshot of every venue's score (for the sparkline)
- `editorial_boost_log` — audit log of every boost applied
- `lists` — the canon lists (dynamic, generated from scores + filters)
- `bnt_awards` — Best New Table winners, editorial pathway
- `pitch_submissions` — reader contributions queue
- `editors` — currently just Martin and Yahnee

## When to stop and ask

Claude Code should stop and ask Martin or Yahnee before:

- Adding any feature that affects how scores are calculated or displayed
- Introducing named critics, critic profiles, or critic-specific UI
- Adding paid promotion, sponsored content, or affiliate integration beyond what's specified in `/docs/product/spec-v2.md`
- Publishing any venue-facing content below 5.0
- Adding new list categories that aren't in `/docs/launch/canon-lists.md`
- Accepting any change that removes a hard rule above
- Changing the algorithm weights, the boost limit, or the data thresholds

When uncertain whether a change touches these areas, ask. Over-asking is cheap; silently introducing a change that breaks the project's credibility is not.
